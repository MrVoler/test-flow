1. 需求 Review

策划完成的需求文档由开发、测试、运营进行 Review，提出 Review 意见并做出修改。通过 初步 Review 发现需求的漏洞并提早改掉，提高之后整个开发过程的效率。

2. 测试设计

2.1 测试用例编写

测试用例是测试设计的一个产出物，它直接体现测试设计的思想，一份漂亮的测试用例不仅仅是设计思路的优，更是便于流转和执行，具有可读性、传递性。

用例模板:

用例编号 用例标题 预置条件 优先级 测试步骤 预期结果 是否通过 备注
项目名-模块名-001 功能模块简报 测试用例执行前提的条件 紧急/正常/较弱 尽量简洁 期望结果 ✅ 注意事项
例:上海天气-天气预警-001 天气预报->天气预警 气象局有天气预警发布时 正常 点击预警按钮-进入预警地图 有预警时正常显示预警并显示正确区县的预警信息 ❌ 只有有预警才出现

参考文档

如何写一份漂亮的测试用例？

2.2 测试场景设计

测试场景设计主要也就是测试环境问题,设计时应考虑到尽可能完备的情景.如不同浏览器和浏览器的版本/不同网络状态等,应尽量列出并在开发时做好相应的容错。

参考文档

测试用例设计白皮书－－场景设计方法

3. 测试用例&场景 Review

参与测试设计人员针对需求写出粗略的用例点与场景之后，可再让策划、开发、测试、运营 Review 一遍，目的还是发现需求的遗漏点，由于测试设计人员已经思考了很多测试点，相当于是对需求进行了细化和剖析，这个 Review 环节还是能发现很多需求的漏洞。

4. 开发自测

4.1 单元测试

集中对用代码实现的每一个程序单元进行测试，检查各个程序模块是否正确地实现了规定的功能。可以借助一些自动化测试工具提升效率。

4.2 确认测试

检查已实现的软件是否满足了需求规格说明中确定了的各种需求，以及代码配置是否完全、正确。

4.3 集成测试

把已测试过的模块组装起来，测试是否与其他模块有冲突。

5. 验收测试

5.1 功能测试

是否完成所有功能模块与需求.

5.2 性能测试

如服务器的压力测试/网页加载速度等.

5.3 安全测试

前后端 Review 部分关乎安全性模块,做出安全评估及改进意见.

5.4 兼容性测试

提升在不同使用环境下产品的稳定性,并在开发进行时就应根据客户的需求兼容性作出相应考虑.

6. 验收测试报告

上线以前，应根据验收测试的结果整理一份报告，重点指出测试过程中发现的问题、及上线以后可能会出的质量问题，并在项目群或者召集会会议把这些风险与相关人员一一沟通过。

7. 线上 Bug 处理系统

7.1 线上 Bug 采集

  • 1. 线上自测出现的 Bug 处理与反馈.
  • 2. 客户及用户反馈的问题.
  • 3. 使用第三方监控工具对产品多次出现 exception 的地方进行还原并分析原因.

7.2 线上 Bug Review

对于线上发现的Bug，应建立完整的 Bug 分析流程，重点分析这个线上 Bug 产生的原因、线上 Bug 的影响范围,然后应根据分析结果在最快时间内做出 Bug 解决方案。此线上 Bug Review 流程应是一个不断循环完善的过程,但此过程应该是朝着高效率低时耗的方向不断发展进步。

8. 测试总结

8.1 测试流程总结

  • 总结该次测试流程的完整度和严谨性,对其中影响过效率的部分应作出思考和改进.
  • 对测试流程进行后效率提升的地方做出记录和总结.

8.2 开发流程总结

  • 在开发过程中遇到的有代表性的 Bug 和解决方案作为案例分析,内部分享交流,避免同类 Bug 再犯.
  • 对开发自测的环节中自动化测试的经验与线上 Bug 的收集解决方案进行总结,提高整体团队测试与开发效率.

9. 产品测试原则

在保障研发效率的前提下提高产品质量。
